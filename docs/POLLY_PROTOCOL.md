# 🗳️ Polly Protocol Spec v1

**Objective**: Define the rules, cryptographic guarantees, and transparency mechanisms for the Omni-Social verifiable polling system.


## 🛡️ 1. Authentication & Identity

- **The Brain (`did\_rust`)**: All cryptographic operations must utilize the internal `did\_rust` library (via FFI or WASM). No external DID libraries (e.g., didkit) are permitted.

- **Mandate**: All users must authenticate via **Decentralized Identifiers (DID)**.

- **Handshake**: Authentication is bridged via the `iyou\_idp` using OpenID Connect (OIDC).

- **Primary Identifier**: The OIDC `sub` claim must contain the user's full DID string (e.g., `did:key:z6Mk...`).

- **Sovereignty**: Legacy password-based authentication is strictly disabled. Identity is proven by the holder's ability to sign a UUID challenge.

## 📜 2. The Cryptographic Ledger

- **Mandate**: Every vote is an immutable event, not just a database entry.

- **Signature-Strict Voting**: A vote **must** be rejected if it does not contain a valid cryptographic signature generated by the user's DID key.

- **Ledger Anchoring**:

  - **Merkle Roots**: The system calculates a SHA-256 Merkle root of vote signatures.

  - **Frequency**: Roots must be generated every 100 votes or upon poll closure.

  - **Command**: Anchoring is performed via the `anchor\_ledger` management command.

- **Storage**:

  - **Local**: Metadata is stored in Django (Postgres/SQLite).

  - **Global**: Finalized ledgers and Merkle roots are published to **IPFS/Blossom** for public auditing.

## ⚙️ 3. Voting Power & Transparency

- **Default Rule**: **1:1** (One user, one vote).

- **Weighted Voting**:

  - Custom ratios (e.g., `investor\_share`) must be declared in poll metadata using `vote\_power\_rule` and `vote\_power\_ratio`.

- **Visual Mandate**:

  - Any poll utilizing a non-1:1 ratio **must** display the gear icon ⚙️.

  - Tooltips must be present to explain the voting power calculation to the user.

## 🛰️ 4. Federation & The Omni-Social Mesh

- **Cross-Instance Polls**: Polly supports **ActivityPub** for federating polls across different instances.

- **Shared Audits**: Federated instances must publish their vote ledgers to a shared network (IPFS) to allow for cross-instance verification.

- **Global Scoping**: Global-scoped polls are aggregated by the `iyou\_wun` satellite app, while local-scoped polls remain within regional instances.

## 🔌 5. API & Integration

### REST Endpoints

| **Endpoint** | **Method** | **Description** |
| - | - | - |
| `/api/polls/` | POST | Create a poll (declares ratios/rules). |
| `/api/polls/\{id\}/cast/` | POST | Cast a signed vote. Requires `signature`. |
| `/api/polls/\{id\}/results/` | GET | Returns results with the anchored Merkle root. |
| `/api/polls/\{id\}/verify/` | GET | Generates the audit-ready JSON for `iyou\_home`. |

### The "Verify on Desktop" Bridge

- Polly provides a "Verify on Desktop" mechanism.

- It generates a signed JSON blob of the vote history which is passed to the **Vault** (`iyou\_home`) via the local WebSocket bridge for independent local verification.

## 🔒 6. Security Guarantees

- **Anti-Tampering**: Any modification to a vote signature will invalidate the Merkle root, triggering automated system alerts.

- **Sovereign Audit**: Users do not need to "trust" the Polly server; they can recalculate the Merkle root on their own machine to verify the integrity of the poll.
